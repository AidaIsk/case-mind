# ARCHITECTURE.md — Реестр архитектурных решений CaseMind

> В условиях роста объёма кейсов и ужесточения требований регуляторов к explainability AI-систем CaseMind обеспечивает прозрачный, проверяемый и воспроизводимый процесс принятия KYC/AML-решений.

> Документ предназначен для аудитора системы, разработчика, входящего в проект, compliance-офицера и команд KYC/AML, которым важно обеспечить консистентность решений. Он объясняет, как именно ИИ участвует в принятии решений и где проходят границы его полномочий.

---

## 1. Цель проекта

CaseMind — система поддержки принятия решений (Decision Support System) для KYC/AML-аналитиков.

**Что это даёт бизнесу:**
- снижение ошибок аналитиков за счёт жёстких логических блокировок
- стандартизация решений: каждый кейс оформляется по единой структуре вне зависимости от аналитика
- ускорение onboarding и периодических пересмотров: AI берёт на себя черновую работу по структурированию, аналитик фокусируется на суждении

Система не принимает решения вместо аналитика. Она помогает аналитику структурировать рассуждение, выявлять пробелы в досье и формировать защищаемую Аналитическую записку (Decision Note), которая фиксирует ход мысли и основание решения.

**Главный принцип:** *ИИ помогает рассуждать, но не заменяет суждение аналитика.*

**CaseMind формализует процесс принятия решений аналитика и делает его проверяемым, воспроизводимым и обучаемым.**

Применимость: KYC onboarding, периодические пересмотры (review), триггерные проверки. Соответствует принципу «Суть над формой» (Substance over Form) — экономическая логика важнее формального наличия бумаг.

> **Нормализация терминов:** в этом документе **Decision Note** и **Аналитическая записка** используются как взаимозаменяемые понятия и обозначают одно и то же: структурированный compliance-документ, формируемый системой по итогам проверки кейса.
>
> Типовая структура документа: Summary → Key Risk Factors → CDD Assessment → Signal Trace → Decisive Factor → Self-Review → Decision Rationale → Next Steps.

## 2. Технологический стек

| Компонент | Инструмент | Роль |
|---|---|---|
| Интерфейс | Python / Streamlit | Многовкладочный UI без бизнес-логики |
| Оркестрация | `services.py` | Единая точка входа для UI |
| Бизнес-логика | `logic.py`, `validators.py` | Жёсткие правила до генерации |
| Интеллектуальный слой | LLM API (OpenAI, GPT-5.4) | Структурированная генерация JSON |
| Валидация вывода | `output_schema.py` | Проверка формы и смысла ответа модели |
| Рендеринг | `renderers.py` | Сборка читаемой Аналитической записки |
| Хранение | `storage.py` / `cases.json` | Персистентность без тяжёлой СУБД |
| Аналитика качества | `learning.py` | Агрегация ошибок по истории кейсов |

**Архитектурный принцип:** строгое разделение слоёв. UI обращается только к `services.py`. `services.py` координирует все остальные модули. Ни один UI-файл не импортирует `logic`, `storage`, `schemas` или `llm` напрямую.

```
UI (new_case, case_view, case_list)
        ↓
    services.py
        ↓
logic / validators / llm / output_schema / renderers / storage
```

**Processing Flow**

```
UI (Streamlit)
    ↓
services.py  —  orchestration layer
    ↓
logic / validators  —  pre-checks (до генерации)
    ↓
llm  —  structured generation (JSON)
    ↓
output_schema  —  validation (форма + логика + смысл)
    ↓
renderers  —  Decision Note (Markdown)
    ↓
storage  —  cases.json
```

Каждый шаг выполняет отдельную функцию и не смешивается с другими слоями. Это обеспечивает прозрачность, тестируемость и explainability системы.

## 3. Логика принятия решений

### 3.1 Три режима исхода

Система различает три принципиально разных исхода — они не взаимозаменяемы:

| decision_mode | Значение | Когда применяется |
|---|---|---|
| `approve` | Одобрить | CDD завершён, риск приемлем, блокирующие факторы отсутствуют |
| `edd` | Эскалация (EDD) | CDD не завершён, но пробелы закрываемы через усиленную проверку |
| `reject` | Отказать | CDD невозможно завершить **или** риск неприемлем даже при завершённом CDD |

### 3.2 Статус CDD как основание режима

`cdd_status` — не просто метка, а обязательная предпосылка режима. Каждый режим имеет ровно одно допустимое значение:

```
decision_mode = "edd"
    → cdd_status = "Incomplete"
    → Пробелы существуют, но могут быть закрыты документально

decision_mode = "reject" + reject_reason_type = "CDD_FAILURE"
    → cdd_status = "Incomplete and cannot be completed"
    → UBO не может быть установлен в принципе — структурный blocker

decision_mode = "reject" + reject_reason_type = "RISK_UNACCEPTABLE"
    → cdd_status = "Complete but risk not acceptable"
    → CDD формально завершён, но риск-сигнал (например, неснятый adverse media) делает клиента неприемлемым

decision_mode = "approve"
    → cdd_status = "Complete"
    → Все элементы CDD подтверждены, существенных блокирующих факторов не выявлено
```

Любое несоответствие между этими полями блокирует сохранение кейса.

### 3.3 Decisive Factor — мост между анализом и решением

`decisive_factor` — единственная формулировка главного перевешивающего фактора. Это то, что аналитик мог бы зачитать на кредитном или compliance-комитете.

Правило формулировки по режиму:

- **EDD:** называет конкретный незакрытый пробел, из-за которого нельзя одобрить, но можно продолжить проверку. Пример: *«Источник средств по операции не подтверждён.»*
- **Reject / CDD_FAILURE:** называет структурный blocker, делающий завершение CDD невозможным. Пример: *«Бенефициарный владелец не установлен и не может быть подтверждён.»*
- **Reject / RISK_UNACCEPTABLE:** называет конкретный неустранённый риск-сигнал. Пример: *«Негативные публикации о возможной вовлечённости в сомнительные схемы не были сняты.»*
- **Approve:** называет главный подтверждающий фактор. Пример: *«Ключевые элементы CDD подтверждены, критические ограничения не выявлены.»*

`decisive_factor` обязан совпадать по смыслу с тем сигналом в `signal_trace`, которому присвоен вес `DECISIVE`. Это требование проверяется автоматически.

### 3.4 Signal Trace — прозрачность аналитической логики

`signal_trace` — список 2–6 конкретных наблюдений, каждое из которых описывает один факт, повлиявший на решение. Общие выводы («совокупность факторов») запрещены — только конкретные наблюдения.

Каждый сигнал содержит:
- `signal` — формулировка наблюдения на русском языке
- `category` — тип: `CDD | SCREENING | GEOGRAPHY | SOF | ECONOMIC_RATIONALE | PROFILE_MISMATCH | OTHER`
- `impact` — вес: `LOW | MEDIUM | HIGH | DECISIVE`
- `direction` — куда указывает: `SUPPORTS_DECISION | SUPPORTS_ESCALATION | SUPPORTS_REJECT | MITIGATING`
- `comment` — одно предложение о том, почему этот сигнал важен

Ровно один сигнал в каждом кейсе должен иметь вес `DECISIVE`. Именно этот сигнал по смыслу должен совпадать с `decisive_factor`.

### 3.5 Жёсткие правила до генерации (Logic Layer)

Следующие проверки выполняются **до** обращения к языковой модели. Нарушение любого из них блокирует генерацию:

| Правило | Основание |
|---|---|
| UBO не установлен → генерация заблокирована | Невозможно завершить CDD без идентификации бенефициара |
| Approve + SoF не подтверждён → заблокировано | RULE 1: нельзя одобрять при неизвестном источнике средств |
| Approve + документы отсутствуют → заблокировано | CDD не может считаться завершённым |
| Approve + экономический смысл не подтверждён → заблокировано | Substance over Form: формальное одобрение без экономической логики недопустимо |
| Approve + уровень риска «Высокий» → заблокировано | Несовместимые значения |
| High-risk geography + уровень риска «Низкий» → заблокировано | Внутреннее противоречие в оценке |

## 4. Валидация вывода языковой модели

### 4.1 Зачем валидировать ответ модели

Языковая модель может вернуть текст, который выглядит убедительно, но содержит логические противоречия — например, написать, что "CDD не может быть завершён" в кейсе, по которому выбрана эскалация (EDD), а не отказ. Для аудитора это важно: система не принимает вывод модели на веру, а проверяет его соответствие compliance-логике.

### 4.2 Три уровня проверки

**Уровень 1 — Форма.** Проверяется наличие всех обязательных полей structured output, определённых в `output_schema.py`, и корректность enum-значений. Неизвестное значение в любом перечислимом поле является ошибкой.

**Уровень 2 — Смысловая консистентность полей решения.** Проверяется, что поля не противоречат друг другу:

- `decision_mode` = "edd" → `cdd_status` обязан быть "Incomplete", `edd_required` = "Да", `reject_reason_type` = "NONE"
- `decision_mode` = "reject" + `reject_reason_type` = "CDD_FAILURE" → `cdd_status` обязан быть "Incomplete and cannot be completed"
- `decision_mode` = "reject" + `reject_reason_type` = "RISK_UNACCEPTABLE" → `cdd_status` обязан быть "Complete but risk not acceptable"
- `decision_mode` = "approve" → `cdd_status` = "Complete", `edd_required` = "Нет", `reject_reason_type` = "NONE"

**Уровень 3 — Смысловая корректность аналитической логики.** Это ключевой уровень для explainability. Проверяется:

*Decisive Factor:*
- Для EDD: `decisive_factor` не должен содержать формулировки о невозможности завершить CDD — это противоречит природе EDD (пробелы закрываемы)
- Для Reject/CDD_FAILURE: `decisive_factor` обязан называть конкретный CDD-blocker (UBO, идентификация, документы)
- Для Reject/RISK_UNACCEPTABLE: `decisive_factor` обязан называть конкретный риск-сигнал (негативные публикации, репутационный риск)

*Signal Trace:*
- Хотя бы один сигнал обязан иметь `impact = DECISIVE`
- `decisive_factor` обязан совпадать по смыслу с текстом DECISIVE-сигнала
- EDD-кейс не может содержать сигнал о невозможности завершить CDD
- Кейс с `RISK_UNACCEPTABLE` не может содержать сигналы о неполноте CDD (UBO не установлен и т.д.) — CDD к этому моменту считается завершённым
- Approve-кейс не может содержать сигналы с `direction = SUPPORTS_REJECT`

*Self-Review:*
- `confidence_score = 5` запрещён при наличии `error_type != NONE` — нельзя одновременно фиксировать ошибку и заявлять максимальную уверенность
- `confidence_score = 5` запрещён при `cdd_status = "Incomplete"` — неполный CDD не может давать максимальную уверенность
- `OVER_REJECT` применим только при `decision_mode = "reject"` — нельзя диагностировать избыточный отказ там, где отказа нет
- `UNDER_REJECT` запрещён при `decision_mode = "reject"` — если решение уже отказ, занижение тяжести семантически невозможно

### 4.3 Поведение при сбое валидации

Если вывод модели не проходит валидацию, система использует технический fallback-ответ. Кейс помечается `error_type = WEAK_RATIONALE`, `confidence_score = 2` и содержит явное указание на необходимость ручной проверки. Fallback-ответ сам по себе проходит schema validation — система никогда не сохраняет структурно некорректную запись.

### 4.4 Типы аналитических ошибок (error_type)

Система фиксирует не только факт ошибки, но и её тип — для последующего обучения аналитика:

| Тип | Значение |
|---|---|
| `NONE` | Существенных аналитических ошибок не выявлено |
| `OVER_REJECT` | Отказ избыточно жёсткий; пробелы, вероятно, закрываемы через EDD |
| `UNDER_REJECT` | Серьёзные блокирующие факторы или риск-сигналы недооценены |
| `MISSED_SIGNAL` | Важный red flag не отражён в анализе или decisive_factor |
| `WEAK_RATIONALE` | Решение допустимо, но обоснование расплывчато и незащищаемо |
| `CDD_LOGIC_GAP` | Перепутаны «пробел в данных» и «невозможность завершить CDD» |
| `INCONSISTENT_DECISION` | Поля решения внутренне противоречат друг другу |

### 4.5 Границы полномочий языковой модели

Языковая модель (LLM) используется исключительно как инструмент структурирования аналитической логики, но не как источник решений.

**Ограничения модели:**

- LLM не изменяет входные данные (`case_data`) — она получает их на вход в неизменном виде
- LLM не принимает окончательное решение вне заданной схемы — допустимые значения `decision_mode` зафиксированы в `output_schema.py`
- LLM не может обойти Logic Layer — жёсткие правила в `logic.py` и `validators.py` отрабатывают до генерации и блокируют неправомерные сценарии независимо от модели
- LLM не может сохранить результат без прохождения валидации — `output_schema.py` является обязательным фильтром между генерацией и хранением

**Все выходные данные модели проходят обязательную проверку:**

- структурную — наличие всех полей, корректность enum-значений
- логическую — консистентность между `decision_mode`, `cdd_status`, `reject_reason_type`
- смысловую — соответствие `decisive_factor`, `signal_trace` и `self_review` заявленному режиму и друг другу

Таким образом, LLM является подконтрольным компонентом системы, работающим в жёстко ограниченной системе правил, а не автономным источником решений. Окончательное решение остаётся за аналитиком.

## 5. Агрегация качества решений

Модуль `learning.py` накапливает сигналы качества по истории кейсов и формирует диагностику системной слабости аналитика. Это переход от оценки отдельного кейса к выявлению паттернов:

- **Error rate** — доля кейсов с `error_type != NONE`
- **Распределение по типам ошибок** — что именно идёт не так
- **Средний confidence score** — общий уровень уверенности в ходе рассуждения
- **Разбивка по режимам** — в каком типе кейсов (EDD / Reject / Approve) концентрируются ошибки

Пример диагноза: *«Слабое обоснование решений (WEAK_RATIONALE) — 88% ошибочных кейсов. Концентрация в EDD/Escalation.»* Это означает, что решения об эскалации в целом верные, но не содержат достаточно конкретного обоснования, чтобы выдержать compliance-проверку.

**Как используются результаты агрегации:**
- адресные подсказки аналитику: система показывает, в каком именно типе кейсов его reasoning стабильно слабеет
- приоритизация обучения: если EDD-кейсы дают WEAK_RATIONALE в 80% случаев, именно туда направляется тренировочная нагрузка
- адаптация кейсов: демо-история (`seed_cases.py`) обновляется с учётом реальных паттернов ошибок

Таким образом, CaseMind не просто фиксирует решения — он постепенно становится адаптивной обучающей системой для compliance-аналитиков.

## 6. Структура хранимой записи

Каждый кейс сохраняется как единый JSON-объект с двумя уровнями:

**Быстрые поля верхнего уровня** — для списка кейсов и агрегации, без необходимости читать весь кейс:

```json
{
  "saved_at": "YYYY-MM-DD HH:MM",
  "case_id": "string",
  "decision": "Одобрить | Отказать | Эскалация",
  "risk_level": "Низкий | Средний | Высокий",
  "decisive_factor": "string",
  "error_type": "NONE | ...",
  "confidence_score": 4
}
```

**Полные данные** — вложенные объекты для детального просмотра:

```json
{
  "case_data": { ... },
  "structured_output": {
    "decision_mode": "edd | reject | approve",
    "cdd_status": "...",
    "signal_trace": [ ... ],
    "decisive_factor": "...",
    "self_review": { ... }
  },
  "decision_note": "Markdown",
  "rejection_reasons": [ ... ],
  "required_actions": [ ... ],
  "timeline": [ ... ]
}
```

## 7. Глоссарий

| Термин | Определение |
|---|---|
| **Decision Note (Аналитическая записка)** | Структурированный документ, фиксирующий ход compliance-рассуждения и основание решения. «Юридический щит» аналитика |
| **CDD (Customer Due Diligence)** | Процедура надлежащей проверки клиента |
| **EDD (Enhanced Due Diligence)** | Усиленная проверка; применяется когда CDD не завершён, но пробелы закрываемы |
| **UBO (Ultimate Beneficial Owner)** | Конечный бенефициарный владелец |
| **SoF (Source of Funds)** | Подтверждённый источник средств |
| **Decisive Factor** | Одна формулировка главного перевешивающего фактора решения |
| **Signal Trace** | Структурированный список наблюдений с весами, объясняющий логику решения |
| **Substance over Form** | Приоритет экономической логики над формальным наличием документов |
| **Logic Layer** | Слой жёстких правил, работающий до языковой модели |
| **Regulatory Override** | Блокировка неправомерного решения системой вне зависимости от выбора аналитика |
